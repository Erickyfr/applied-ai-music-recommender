How Streaming Platforms Predict What You'll Love
==================================================

This project is a hands-on simulation of the core ideas used by real platforms.
Here's how it maps:


Content-Based Filtering — "What does the *song* sound like?"
-------------------------------------------------------------

This is the approach this project uses. It recommends songs by comparing song
attributes to the user's taste profile.

The recommender.py file already has the data structure for this:

  Song Feature     | What it captures
  -----------------|----------------------------------
  genre, mood      | Categorical identity
  energy, tempo_bpm| How fast/intense it feels
  valence          | Emotional positivity
  danceability     | Rhythmic groove
  acousticness     | Organic vs. electronic sound

Spotify does the same thing — their audio analysis pipeline extracts these exact
features from raw audio waveforms. When it recommends a song, it's partly asking:
"Does this song's audio fingerprint match what this user typically likes?"

Strength: Works even for brand-new songs with no listening history.
Weakness: Can get stuck in a "bubble" — if you like chill lofi, it keeps
          recommending chill lofi forever.


Collaborative Filtering — "What do *similar users* like?"
----------------------------------------------------------

This is what this project does NOT simulate, but it's how Spotify's Discover
Weekly actually works.

Instead of looking at song attributes, it looks at user behavior across millions
of listeners:

  "Users who listened to the same 8 songs as you also tend to love this one
   song you've never heard."

It builds a giant matrix of (user x song) play counts, then finds patterns.
No audio analysis needed — it doesn't even care what the song sounds like.

Strength: Can surface surprising, serendipitous discoveries across genres.
Weakness: Suffers from the cold start problem — new songs and new users have no
          history, so it can't make good predictions for them.


The Key Difference
------------------

                   | Content-Based                      | Collaborative
-------------------|------------------------------------|---------------------------
Data source        | Song attributes (CSV columns)      | Other users' listening behavior
Used in project    | Yes — UserProfile vs. Song features| No
Real-world example | Spotify audio features, Pandora    | Spotify Discover Weekly,
                   | Music Genome Project               | YouTube "Up Next"
Cold start problem | Handles new songs well             | Struggles with new songs/users
Discovery potential| Low (stays in comfort zone)        | High (crosses genre lines)


What Real Platforms Do
----------------------

Spotify, YouTube, and Apple Music use hybrid systems — both at once:

  1. Content-based to handle new songs and establish baselines
  2. Collaborative to find surprising cross-user patterns
  3. Contextual signals (time of day, device, activity) layered on top

This project is a clean, transparent version of step 1 — and that's exactly
what makes it a great foundation for understanding the full picture.





Main Data Types in Recommendation Systems
==========================================

Both collaborative and content-based filtering rely on different categories of
data. Here are the key types used in real systems and how they map to this project:


1. Behavioral Data (used in Collaborative Filtering)
-----------------------------------------------------
This is data generated by what users *do*, not what a song sounds like.

  - Likes / Thumbs Up     : Explicit signal that a user enjoys a song
  - Skips                 : Strong negative signal — user didn't want to hear it
  - Replays               : A skip in reverse — high engagement indicator
  - Saves / Adds to Library: Long-term preference signal
  - Playlist adds         : User curated this song into a context (e.g. "workout")
  - Listen duration       : Did the user finish the song or bail at 10 seconds?
  - Share events          : User found it worth sending to someone else

  These are NOT stored in this project's songs.csv — they require multiple users
  and session tracking over time.


2. Audio / Content Features (used in Content-Based Filtering)
--------------------------------------------------------------
These are measurable properties of the song itself, extracted from the audio.

  - tempo_bpm             : Speed of the song in beats per minute (e.g. 78 = slow, 152 = fast)
  - energy                : How intense or loud the track feels (0.0 to 1.0)
  - valence               : Emotional tone — high = happy/upbeat, low = sad/dark (0.0 to 1.0)
  - danceability          : How suitable the rhythm is for dancing (0.0 to 1.0)
  - acousticness          : How acoustic (vs. electronic/produced) the track sounds (0.0 to 1.0)

  All five of these are columns in this project's data/songs.csv.


3. Categorical / Metadata Features
------------------------------------
These are labels assigned to a song, often by humans or classifiers.

  - genre                 : Broad style category (pop, rock, lofi, jazz, ambient, etc.)
  - mood                  : Emotional label (happy, chill, intense, moody, focused, etc.)
  - artist                : Identity of the creator — used for artist affinity modeling
  - release year          : Recency can affect recommendation weight

  genre and mood are both in this project's Song dataclass and songs.csv.


4. User Preference Data
------------------------
This is what the system knows (or infers) about a specific user.

  - favorite_genre        : Stored directly in this project's UserProfile
  - favorite_mood         : Stored directly in this project's UserProfile
  - target_energy         : Numeric preference for intensity, stored in UserProfile
  - likes_acoustic        : Boolean preference for acoustic vs. produced sound, in UserProfile

  In real platforms, these are not stored explicitly — they are *inferred* from
  behavioral data (likes, skips, replays) over time.


5. Contextual Data (used in advanced hybrid systems)
-----------------------------------------------------
Real platforms also factor in *when and how* a user is listening.

  - Time of day           : Morning commute vs. late night session
  - Device type           : Phone (likely mobile/active) vs. desktop (likely focused work)
  - Activity context      : Workout playlist vs. sleep playlist
  - Location              : Home, gym, car, etc.

  This project does not use contextual data, but it's a natural next step for
  future improvements noted in the model card.

Feature Effectiveness Analysis for Content-Based Recommending
==============================================================

Based on the 10 songs and 10 attributes in data/songs.csv:


Tier 1 — Most Effective (High Signal, Easy to Match)
-----------------------------------------------------

genre
  The single strongest filter. A user who wants lofi will almost never want
  rock in the same session. Genre cleanly separates songs into distinct
  listening contexts with no overlap. Use it as a hard filter or heavy
  multiplier.

mood
  Closely tied to genre in this dataset but captures *intent* better. A user
  doesn't always know they want "lofi" — but they know they want "chill." Mood
  bridges genre labels and user feelings. Very effective as a primary match
  signal.

energy
  The most numerically expressive feature in the CSV. It spans the full range
  (0.28 to 0.93) with good distribution across the catalog. Easy to compute
  distance from a user's target_energy. A user going to the gym and a user
  studying have completely different energy needs — this captures that gap
  precisely.


Tier 2 — Effective as Secondary Signals
----------------------------------------

acousticness
  Strongly inversely correlated with energy in this dataset, but it adds
  something energy doesn't: texture. Two songs can have similar energy levels
  but feel completely different if one is a live guitar track and the other is
  a synthesizer. The likes_acoustic boolean in UserProfile already maps to
  this.

valence
  This is the most emotionally nuanced feature. Energy tells you intensity —
  valence tells you direction (happy vs. dark). A high-energy sad song (think
  intense/moody) and a high-energy happy song are very different experiences.
  Valence separates them. Underused in most beginner recommenders but highly
  effective.


Tier 3 — Useful But Weaker Alone
----------------------------------

tempo_bpm
  Correlated with energy but not identical. Useful for fine-tuning within a
  genre (e.g. a user who specifically wants "slow jazz" vs. "uptempo jazz").
  In a 10-song catalog, the signal is a bit redundant since energy already
  distinguishes fast from slow tracks. Better as a tiebreaker than a primary
  feature.

danceability
  The most context-dependent feature. Its meaning changes completely by genre —
  high danceability in pop means something different than in jazz. Useful for
  playlist-context recommending (e.g. "party mode") but noisy as a standalone
  signal.


Do These Features Match How "Vibe" Actually Works?
---------------------------------------------------

Where the features align well:

  The combination of mood + energy + valence does a strong job of capturing
  the core of what a "vibe" is. When you pick music for a late-night drive vs.
  a morning run, you're really adjusting all three of those axes simultaneously
  — and those three features can represent that mathematically.

  acousticness also genuinely matters to lived experience. The difference
  between a lo-fi bedroom track and a polished studio pop track with the same
  BPM is felt immediately, and acousticness captures that.

Where the features fall short:

  Real vibe is partly about familiarity and association — a song tied to a
  specific memory hits differently regardless of its audio features. None of
  the CSV columns can capture that. That's exactly the gap collaborative
  filtering fills.

  genre labels are also blunt instruments. "Indie pop" and "pop" in this
  catalog have very similar energy and valence profiles — the label distinction
  matters less than the numbers. Real listeners define vibe by feel, not genre
  tag.

  The missing feature: lyrical content. A song with dark lyrics but upbeat
  instrumentation (high valence, high energy) would confuse a purely
  feature-based recommender. None of the CSV attributes touch language or
  lyrical theme at all.


Recommended Feature Priority for Your Recommender
---------------------------------------------------

  Primary match:   genre (exact match bonus) + mood (exact match bonus)
  Numeric scoring: energy distance from target_energy  <- most impactful
  Secondary:       valence (add if user profile includes it)
                   acousticness (already in UserProfile as likes_acoustic)
  Tiebreaker:      tempo_bpm or danceability


Algorithm Recipe: Content-Based Scoring
========================================

Each song gets a total score (0.0 to 1.0) by summing weighted components:

  total_score = genre_score
              + mood_score
              + energy_score
              + acoustic_score
              + valence_bonus


Component Rules
---------------

1. genre_score — weight: 0.35

  if song.genre == user.favorite_genre  ->  0.35
  else                                  ->  0.00

  Hard match. Genre mismatch is the strongest disqualifier.

2. mood_score — weight: 0.25

  if song.mood == user.favorite_mood  ->  0.25
  else                                ->  0.00

  Hard match. Mood is the second strongest filter.

3. energy_score — weight: 0.25

  distance     = abs(song.energy - user.target_energy)
  energy_score = (1.0 - distance) * 0.25

  Continuous match. A song exactly at the user's target energy scores full
  points. A song 1.0 away (max distance) scores 0.

4. acoustic_score — weight: 0.10

  if user.likes_acoustic == True  and  song.acousticness >= 0.6  ->  0.10
  if user.likes_acoustic == False and  song.acousticness <  0.4  ->  0.10
  else                                                            ->  0.00

  Boolean preference mapped to a threshold. Threshold 0.6/0.4 chosen from
  the natural gap observed in songs.csv.

5. valence_bonus — weight: 0.05

  if user.favorite_mood in (happy, relaxed) and song.valence >= 0.7  ->  0.05
  if user.favorite_mood in (moody, intense) and song.valence <= 0.5  ->  0.05
  else                                                                ->  0.00

  Soft tiebreaker. Rewards emotional alignment between mood label and valence.


Score Summary Table
-------------------

  Component    | Max Points | Match Type          | UserProfile Field
  -------------|------------|---------------------|-------------------
  genre        | 0.35       | Exact               | favorite_genre
  mood         | 0.25       | Exact               | favorite_mood
  energy       | 0.25       | Continuous distance | target_energy
  acousticness | 0.10       | Threshold boolean   | likes_acoustic
  valence      | 0.05       | Soft conditional    | favorite_mood
  Total        | 1.00       |                     |


Recommendation Decision
------------------------

  1. Score all songs using the recipe above
  2. Sort by total_score descending
  3. Return top k songs (default k=5)
  4. Explanation = list which components fired for that song


Why These Weights
-----------------

  Genre + mood = 0.60 combined — together they define the session context.
  A mismatch on both is an immediate near-zero score.

  Energy = 0.25 — the most continuous and expressive numeric feature; gives
  fine-grained ranking within a genre/mood match.

  Acousticness = 0.10 — real texture difference but not session-defining
  on its own.

  Valence = 0.05 — a tiebreaker, not a decider. Keeps the scoring
  transparent and avoids over-engineering.

  This recipe is fully implementable with the existing UserProfile and Song
  fields in src/recommender.py — no new data fields needed.
